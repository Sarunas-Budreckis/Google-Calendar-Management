# Story 8.5: Rendering and Drilldowns Mint Candidates in New Model

**Epic:** 8 — Event Model & Raw Data Linking Engine
**Status:** in-progress
**Agent:** Sonnet · **Effort:** medium
**Prerequisites:** Story 8.3 (Event repository + identity service) must be complete

---

## Story

As a developer working on Epic 8,
I want all event creation paths and calendar rendering to use the unified `event` table's `lifecycle`/`publish` fields,
so that drilldown-created events are `lifecycle=candidate` (translucent), manual creates are `lifecycle=approved`, `CalendarEventSourceKind` is derived from lifecycle state, and Outlook events no longer render as calendar events.

---

## Acceptance Criteria

1. All drilldown "Create Candidate Event (autogenerated)" buttons (TogglSleep, Civ5, TogglTransit, ComfyUI, TogglPhone, TogglSleepCompact, CallLog card provider, TogglSleep card provider, TogglTransit card provider) create events with `lifecycle=candidate, publish=local_only` via `IEventRepository`; these events render translucently (Opacity=0.6) on the calendar.
2. Manual event creation paths (drag-to-create in WeekView/DayView, day-click in `DataSourcePanelViewModel`, "new event" button in MainPage) create events with `lifecycle=approved, publish=local_only`; these render with full opacity and the existing local-draft status label.
3. `CalendarQueryService` reads exclusively from the unified `event` table via `IEventRepository`; the `gcal_event`+`pending_event` join query and the `OutlookEvent` query are removed from this service.
4. `CalendarEventSourceKind` enum: `Candidate` value added, `Outlook` value removed. The value is derived inside `CalendarQueryService.MapEventToDisplayModel` from `(lifecycle, publish)`:
   - `lifecycle=candidate` → `Candidate`
   - `lifecycle=approved, publish=published` → `Google`
   - `lifecycle=approved, publish=local_only` → `Pending`
5. Calendar renders three lifecycle states distinctly: `Candidate` (Opacity=0.6), `Pending` (Opacity=0.6, local-draft status label), `Google` (Opacity=1.0; Opacity=0.6 when `has_unpublished_changes=true`).
6. `EventDetailsPanelViewModel` handles `CalendarEventSourceKind.Candidate`: shows "Candidate event" source display, exposes `ApproveCommand` (sets `lifecycle=approved` via `IEventRepository`), routes all save/revert/delete operations through `IEventRepository` (removes direct use of `IPendingEventRepository` and `IGcalEventRepository` for save paths).
7. Outlook events do not render on the calendar. All `CalendarEventSourceKind.Outlook` switch cases and Outlook-prefixed event-id routing are removed from `CalendarQueryService`, `EventDetailsPanelViewModel`, Views, and display models. `OutlookEvent` table and Outlook drilldown/card code are **not touched** (raw data stays; only calendar rendering is removed).
8. All `switch` statements on `CalendarEventSourceKind` in Views and ViewModels handle (or explicitly ignore) `Candidate`; no unhandled enum values remain after removing `Outlook`.
9. Existing unit/integration tests updated for new enum values; new tests cover: `SourceKind` derivation from `lifecycle`/`publish`, candidate Opacity, `ApproveCommand` lifecycle transition, and drilldown VMs producing `Candidate` kind.

### SourceKind derivation logic (goes into `CalendarQueryService.MapEventToDisplayModel`)

```csharp
var sourceKind = ev.Lifecycle == "candidate"
    ? CalendarEventSourceKind.Candidate
    : ev.Publish == "published"
        ? CalendarEventSourceKind.Google
        : CalendarEventSourceKind.Pending;

var opacity = ev.Lifecycle == "candidate" || ev.HasUnpublishedChanges ? 0.6 : 1.0;

var statusLabel = ev.Lifecycle == "candidate"
    ? "Candidate event"
    : ev.Publish == "local_only"
        ? "Not yet published to Google Calendar"
        : ev.HasUnpublishedChanges
            ? "Local changes, pending push to GCal"
            : string.Empty;
```

`IsPending` on `CalendarEventDisplayModel` should remain `true` for both `Pending` and `Candidate` source kinds — many rendering helpers still test `IsPending` as a shortcut. Set it to `ev.HasUnpublishedChanges || ev.Publish == "local_only"` (i.e., true whenever not a clean published event).

### Drilldown pattern — before and after

**Before (8.4 and earlier — PendingEvent table):**
```csharp
var draft = await _pendingEventDraftService.CreateDraftAsync(startLocal, endLocal, "Sleep");
draft.SourceSystem = "toggl";
draft.ColorId = "grey";
await _pendingEventRepository.UpsertAsync(draft);   // ← secondary upsert
WeakReferenceMessenger.Default.Send(new EventUpdatedMessage(draft.PendingEventId));
_calendarSelectionService.Select(draft.PendingEventId, CalendarEventSourceKind.Pending, openInEditMode: true);
```

**After 8.5 (unified Event table):**
```csharp
var candidate = await _pendingEventDraftService.CreateCandidateAsync(
    startLocal, endLocal,
    summary: BuildSleepTitle(_quality),
    sourceSystem: "toggl",
    colorId: "grey");
// No secondary upsert — CreateCandidateAsync writes the complete row and sends EventUpdatedMessage
_calendarSelectionService.Select(candidate.EventId, CalendarEventSourceKind.Candidate, openInEditMode: true);
```

Drop the injected `IPendingEventRepository` from each drilldown VM constructor if the secondary upsert was its **only** use — check carefully; some VMs use it for other things (e.g. `TogglSleepDrilldownViewModel` also uses `IPendingEventRepository.GetSleepEventForDateAsync` in `UpdateSleepEventTitleAsync`).

### EventDetailsPanelViewModel: save path surgery

The VM currently branches on `_currentSourceKind` to route between `IPendingEventRepository` (draft saves) and `IGcalEventRepository` (to look up the GCal event for creating an overlay). After 8.5 the unified `event` table eliminates the dual-table pattern:

| Old operation | New operation |
|---|---|
| `_pendingEventRepository.GetByPendingEventIdAsync` | `_eventRepository.GetByEventIdAsync` |
| `_pendingEventRepository.UpsertAsync(pendingOverlay)` (for Google event edit) | `_eventRepository.UpdateAsync(event)` with `has_unpublished_changes=true` |
| `_gcalEventRepository.GetByGcalEventIdAsync` | Not needed — `event` row is already the source of truth |
| `_pendingEventRepository.DeleteByPendingEventIdAsync` | `_eventRepository.DeleteAsync(eventId)` |
| `_pendingEventRepository.DeleteByGcalEventIdAsync` | `_eventRepository.RevertToLastSyncedAsync(eventId)` (resets `has_unpublished_changes`) |

The `EventId` used by the panel is now always the stable `event_id` string. The panel should no longer interpret the `"pending_"` prefix — all events use their stable `event_id`.

### Candidate in EventDetailsPanelViewModel

- `ApproveCommand` is the primary new action. It calls `IEventRepository.UpdateAsync(event with { Lifecycle = "approved" })` then refreshes.
- `ApproveButtonVisibility` visible when `_currentSourceKind == Candidate`.
- Candidates can be edited (title, time, color) identically to pending drafts. Edit behavior is the same.
- Candidates cannot be pushed to GCal directly (no "Push" / "Ready to Publish" toggle until approved).
- On delete of a Candidate: direct `IEventRepository.DeleteAsync` (no staging, no GCal removal needed).
- `AllowsBlankDraftTitle` should include `Candidate` — newly minted candidates often have a pre-filled title but allowing blank is fine.
