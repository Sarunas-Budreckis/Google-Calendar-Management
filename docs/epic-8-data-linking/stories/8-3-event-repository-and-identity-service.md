# Story 8.3: Event Repository + Identity Service

**Epic:** 8 — Event Model & Raw Data Linking Engine
**Status:** done
**Agent:** Opus · **Effort:** medium
**Dependencies:** 8.2 (blocking — unified `event` table + `Event` entity must exist)

---

## Story

As the application's service layer,
I want a proper `EventRepository` and `IEventIdentityService` over the unified `event` table,
so that all event creation, lookup, editing, deletion, and publishing uses a stable `event_id` and the correct `has_unpublished_changes` / `lifecycle` / `publish` semantics — replacing the now-stubbed `PendingEventRepository`, `PendingEventDraftService`, and `PendingEventPublishService` that 8.2 marked `NotImplementedException`.

---

## Acceptance Criteria

1. `IEventRepository` interface exists in `Services/` with methods:
   - `GetByEventIdAsync(string eventId, CancellationToken ct)` → `Event?`
   - `GetByGcalEventIdAsync(string gcalEventId, CancellationToken ct)` → `Event?`
   - `GetByDateRangeAsync(DateOnly from, DateOnly to, CancellationToken ct)` → `IList<Event>` (non-deleted, non-candidate events (non-autogenerated) in range)
   - `UpsertAsync(Event ev, CancellationToken ct)` → saves or updates by `event_id`
   - `DeleteByEventIdAsync(string eventId, CancellationToken ct)` → removes row
   - `GetDayNameEventAsync(DateOnly date, CancellationToken ct)` → `Event?` (source_system='day_name')
   - `GetSleepEventForDateAsync(DateOnly date, CancellationToken ct)` → `Event?`
2. `EventRepository : IEventRepository` is implemented in `Services/EventRepository.cs` using `IDbContextFactory<CalendarDbContext>` (same pattern as `GcalEventRepository`).
3. `IEventIdentityService` interface exists with:
   - `MintEventId()` → `string` — generates a new stable `event_id` (same UUID format used since Story 8.2: `Guid.NewGuid().ToString("N")` or the 32-char lowercase hex string)
   - `ResolveEventIdAsync(string? eventId, string? gcalEventId, CancellationToken ct)` → `string?` — given either key, returns the stable `event_id` if found
4. `EventIdentityService : IEventIdentityService` is implemented in `Services/EventIdentityService.cs`. `ResolveEventIdAsync` first tries `event_id` direct lookup, then falls back to `gcal_event_id` lookup.
5. `IPendingEventDraftService` is re-implemented (not just stubbed) to call `IEventRepository.UpsertAsync` and `IEventIdentityService.MintEventId`, creating events with `lifecycle='approved'`, `publish='local_only'`, `has_unpublished_changes=false`, `source_system='manual'`. The returned type is updated to return `Event` (or a new thin adapter record — see Dev Notes).
6. `IPendingEventPublishService.PublishAsync` correctly reads pending events from `context.Events` where `has_unpublished_changes=true` (or `publish='local_only'`), calls GCal API, then on success: sets `publish='published'`, fills `gcal_event_id`, clears `has_unpublished_changes`. `event_id` is never changed.
7. `IPendingEventPublishService.GetPendingItemsAsync` reads from `context.Events` where `has_unpublished_changes=true OR publish='local_only'`, projecting into `PendingPublishListItem` records.
8. `CalendarQueryService.GetEventsForRangeAsync` is rewritten (the `// TODO 8.3+` stub from 8.2 is replaced) to query `context.Events` for the given date range, projecting rows into `CalendarEventDisplayModel`. `CalendarEventSourceKind` derivation: `published` → `Google`, `local_only + approved` → `Pending`, `candidate` → `Pending` (with `Opacity=0.5`). Outlook events remain a separate query path (unchanged).
9. `CalendarQueryService.GetEventByIdAsync` is implemented to query `context.Events` by `event_id`.
10. `IEventRepository` and `IEventIdentityService` are registered in `App.xaml.cs` DI container as `AddSingleton`.
11. The old `IGcalEventRepository` / `GcalEventRepository` are **not deleted** in this story (they may still be referenced by `SyncManager`). They are left in place. Add a `// TODO 8.4: remove after sync reconciler is rewritten` comment on the interface.
12. Integration tests in `GoogleCalendarManagement.Tests/Integration/EventRepositoryTests.cs` cover:
    - Create event via `UpsertAsync`, retrieve by `event_id`
    - Create event, then retrieve by `gcal_event_id`
    - `MintEventId()` returns a non-null, non-empty string that is different on each call
    - `ResolveEventIdAsync` by `event_id` → returns same id; by `gcal_event_id` → returns the stable `event_id`
    - `GetByDateRangeAsync` returns events whose time range overlaps the requested window
    - `DeleteByEventIdAsync` removes the event; subsequent lookup returns null

## Dev Notes

### Build context: what 8.2 left for this story

Story 8.2 created:
- `Data/Entities/Event.cs` — all columns from concepts.md §3
- `Data/Configurations/EventConfiguration.cs` — table `event`, indexes, check constraints
- `CalendarDbContext.Events` DbSet (GcalEvents/PendingEvents removed)
- The migration that transforms old tables into `event`
- **Stubs** in: `PendingEventRepository` (throws `NotImplementedException("Replaced in 8.3")`), `PendingEventDraftService` (same), `PendingEventPublishService` (same), `CalendarQueryService.GetEventsForRangeAsync` (returns empty list with `// TODO 8.3+`)

This story's job is to replace every stub with real implementations.

### `event_id` generation — match the 8.2 contract

The migration used `lower(hex(randomblob(16)))` (SQL) for existing rows. In C# code (new events), use:
```csharp
Guid.NewGuid().ToString("N")  // 32-char lowercase hex string, no hyphens
```
This matches the format used by `pending_event_id` (was `pending_{Guid.NewGuid():N}`) minus the `pending_` prefix. Do NOT add a prefix to `event_id` — the PK is a plain UUID hex string.

### Repository pattern — match `GcalEventRepository`

Follow `GcalEventRepository` exactly:
- Constructor: `IDbContextFactory<CalendarDbContext>` (not `CalendarDbContext` directly — allows concurrent use)
- Each method: `await using var context = await _dbContextFactory.CreateDbContextAsync(ct);`
- Read-only queries: `.AsNoTracking()`
- `namespace GoogleCalendarManagement.Services;`

### `IPendingEventDraftService` return type change

The interface return type changes from `Task<PendingEvent>` to `Task<Event>`. Find all callers using:
```
grep -rn "CreateDraftAsync" Services/ ViewModels/
```
Key callers likely include:
- `ViewModels/` — drag-to-create VM, event creation panel VM
- Any service that receives the result and maps it to a display model

Update callers to use `Event` properties (`EventId`, `Summary`, `StartDatetime`, etc.) instead of `PendingEvent` properties.

### `CalendarQueryService` rewrite — derive `SourceKind` from `lifecycle` + `publish`

The old code had a complex JOIN between `gcal_event` and `pending_event`. The new code is simpler:

```csharp
var events = await context.Events
    .AsNoTracking()
    .Where(e => e.Lifecycle != "candidate" &&
                e.StartDatetime.HasValue &&
                e.StartDatetime < rangeEndExclusiveUtc &&
                (e.EndDatetime ?? e.StartDatetime.Value) >= rangeStartUtc)
    .OrderBy(e => e.StartDatetime)
    .ThenBy(e => e.Summary)
    .ToListAsync(ct);
```

`SourceKind` derivation logic:
```csharp
var sourceKind = e.Publish == "published" ? CalendarEventSourceKind.Google : CalendarEventSourceKind.Pending;
var opacity = e.Lifecycle == "candidate" ? 0.5 : 1.0;
```

Note: `lifecycle='candidate'` events ARE included in the range query for Story 8.3 (they show translucent). Story 8.5 later changes how candidates are rendered, but don't pre-optimize here.

### `PendingEventPublishService` — status label derivation

Old code inferred "new draft" vs "edited event" from whether `GcalEventId` was null. New derivation:
- `publish='local_only' AND lifecycle='approved'` → "new draft"
- `publish='published' AND has_unpublished_changes=true` → "edited event"
- `publish='published' AND has_unpublished_changes=true AND operation_type=...` → check for delete operation (see 8.2 dev notes on `operation_type='delete'` rows)

The `operation_type` field was carried over from `pending_event` during migration. Check `Event.OperationType` or equivalent if mapped.

### `RevertAsync` — handle two cases

1. **Local-only event** (`publish='local_only'`): simple delete from DB — `DeleteByEventIdAsync`.
2. **Published event with unpublished changes** (`publish='published', has_unpublished_changes=true`): the cleanest approach is to re-fetch the event from GCal via `IGoogleCalendarService` and overwrite local fields, then set `has_unpublished_changes=false`. If re-fetch is too complex for this story, add `// TODO 8.4: improve RevertAsync for published events` and implement as: clear `has_unpublished_changes=false` (loses the edit, keeps gcal version on next sync).

### What this story does NOT do

- Does NOT delete `IGcalEventRepository` / `GcalEventRepository` — `SyncManager` still uses it; leave with `// TODO 8.4` comment

- Does NOT change `CalendarEventDisplayModel` shape — Story 8.5
- Does NOT rewrite `SyncManager` / GCal sync reconciler — Story 8.4
