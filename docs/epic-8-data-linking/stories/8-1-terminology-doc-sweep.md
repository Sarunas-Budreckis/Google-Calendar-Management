## Acceptance Criteria

1. No planning doc uses "integration" to mean a data concept (coverage, data source, or datapoint). The word may still appear in non-data contexts (e.g. "integrated solution", "Epic 5 builds the infrastructure", proper noun `DateSourceIntegration` when referring to the old table name in a historical/migration context).
2. No planning doc uses "artifact" to mean a datapoint.
3. Every use of "candidate event" is accompanied by "(autogenerated)" on first use within a doc, or the doc's glossary note clarifies it is autogenerated.
4. Each affected epic overview (`epic-5`, `epic-6`, `epic-7`) has a **Concepts reference** link to `docs/epic-8-data-linking/concepts.md` at the top (same pattern as epic-8 and epic-9 overviews already have).
5. Epic 8 and Epic 9 overviews already use correct terminology — no changes needed to their bodies, only verify.
6. Story files in epics 1–4 are historical/done; sweep for egregious misuse only, do not rewrite completed implementation notes.
7. No behavior changes: this is text-only; no C# code, XAML, migrations, or test files are touched.

## Terminology Replacement Rules

Apply these in order. All replacements are in planning-doc prose only — never touch source code.

### Rule 1 — Retire "integration" for data concepts

| Old phrase (data concept) | Canonical replacement |
|---------------------------|----------------------|
| "integration tracking" | "coverage tracking" |
| "integration checkbox" | "coverage indicator" or "coverage checkbox" |
| "marks a day as integrated" | "marks a day as covered" |
| "source integration" (meaning "source has been accounted for") | "source coverage" |
| "Epic 5 builds the infrastructure (registry, **integration** tracking, drilldown contract)" | "… (registry, coverage tracking, drilldown contract)" |
| `DateSourceIntegration` when used as a concept description | "manual per-day coverage checkbox (deprecated; replaced by computed coverage in Epic 8)" |

**Do NOT replace:**
- "integration" in proper noun code identifiers already tagged for deletion (`DateSourceIntegration` table, etc.) when the sentence is clearly about the old code that will be removed.
- "Epic integrations" or "Epic 5 integration" when it means the epic itself, not a data concept.
- The word "integrated" in non-data-concept contexts ("integrated solution", "tightly integrated").

### Rule 2 — Retire "artifact" for datapoints

| Old | New |
|-----|-----|
| "data artifact" | "datapoint" |
| "source artifact" | "datapoint" or "raw data" (per context) |
| "artifact" used as synonym for a raw data record | "datapoint" |

**Do NOT replace:** "planning artifact", "implementation artifact" (project-management terms, unrelated).

### Rule 3 — Candidate event = autogenerated

On first use of "candidate event" in each planning doc, ensure it reads:
> "candidate event (autogenerated)"

or the section includes a parenthetical clarification. Subsequent uses in the same doc need no qualifier.

### Rule 4 — Add Concepts reference link to epic overviews

Add the following line immediately after the `**Status:**` line (or after `**Tier:**` if present) in each epic overview that does not already have it:

```
**Concepts reference:** [Epic 8/9 vocabulary & data model](../epic-8-data-linking/concepts.md)
```

Epics 8 and 9 already have this line; skip them.

## Dev Notes

### Scope is docs only — zero code changes

This story touches **only** `.md` files under `docs/`. No C#, XAML, migration, test, or config files. If a grep turns up the old terms in source code, **do not fix them here** — they are addressed by the functional Epic 8 stories and by Story 8.16 (closing code terminology cleanup).

### Canonical vocabulary reference

`docs/epic-8-data-linking/concepts.md` is the single source of truth. When in doubt about what term to use, consult §2 (Vocabulary table) of that file.

Key replacements at a glance:
- "integration" (data concept) → "coverage" (for the tracking concept) or "data source" (for the source itself)
- "artifact" (data) → "datapoint"
- "candidate event" → "candidate event (autogenerated)" on first mention
- "pending event" (machine-generated) → "candidate event" in planning text (the code rename happens in Story 8.5)
- "approved event" — already canonical; use consistently

### What "integration" is allowed to mean

The word "integration" is NOT retired universally. It is only retired as a synonym for "data coverage" or "data accounting". These uses are fine and must not be changed:
- "`DateSourceIntegration` table" — historical name of a table being deleted in 8.10; name is a code identifier.
- "Epic 5 builds the infrastructure" — infrastructure is a legitimate word.
- "tightly integrated components" — standard engineering language.

### Epic 7 overview has several occurrences

The Epic 7 overview (`docs/epic-7-data-sources/epic-overview.md`) uses the old pattern extensively in its "Shared Patterns" section. Expected replacements:
- "Integration checkbox in left panel day mode (per-source per-day)" → "Coverage indicator in left panel day mode (per-source per-day)"
- "integration tracking, drilldown contract" (in the Background paragraph) → "coverage tracking, drilldown contract"

### Story files — scope limiter for done stories

Story files for done work (e.g. 7.11 Civ5, 7.10 Spotify) contain Dev Agent Records and implementation notes that describe what was actually built. **Do not edit** those sections — they are historical records. Only edit the Story / Acceptance Criteria / Dev Notes / Tasks sections.
