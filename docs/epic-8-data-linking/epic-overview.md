# Epic 8: Event Model & Raw Data Linking Engine

**Author:** Sarunas Budreckis
**Date:** 2026-06-11
**Status:** Ready for Story Creation
**Tier:** 4 (data accountability layer)
**Concepts reference:** [concepts.md](./concepts.md) — canonical vocabulary + data model

---

## Goal

Build the **foundation and engine** for total raw-data accountability: a unified event
model with a stable local identity, a normalized **datapoint** registry that every data
source projects into, a **link** model (link / ignore / unlink, all undoable), and a
hardcoded **rule engine** that proposes link / ignore / generate-candidate operations while
preserving linkage integrity across any sequence of operations.

The user-facing workflows live in **Epic 9 (Linking Panel)**; Epic 8 is the data + engine
substrate it stands on.

### Phase 0 — Terminology + Event unification

#### Story 8.1 — Terminology doc-sweep across planning artifacts
**Agent:** Codex · **Effort:** low
Rewrite all planning docs (Epics 1–7, `epics.md`, story files) to the canonical vocabulary in
[concepts.md](./concepts.md): *candidate = autogenerated*, retire *integration*, adopt
*clump/block*, *datapoint / raw data*, *Linking panel*, *link order*, *certify day*,
*approved event*. Text only; no behavior change.
**Key AC:** no planning doc uses "integration" for a data concept; "candidate event" defined
as autogenerated everywhere; glossary linked from each affected epic overview.
**Prereqs:** concepts.md (exists).

#### Story 8.2 — Unified `event` table + stable id + atomic migration
**Agent:** Opus · **Effort:** high
Create the unified `event` table (§3 of concepts) with stable `event_id`, `gcal_event_id`
(nullable/unique), `lifecycle`, `publish`, `has_unpublished_changes`. One atomic EF migration
transforms `gcal_event` + `pending_event` → `event`, rewrites `LinkedEventId` → `event_id`.
**Key AC:** every prior event present exactly once; gcal-backed events keep their gcal id in
`gcal_event_id`; overlay pending rows merged (not duplicated) with `has_unpublished_changes`;
machine pending rows become `candidate`; migration is reversible-by-restore (DB backup taken
first per existing Story 1.4 infra); full test coverage on the transform.
**Prereqs:** 8.1 recommended (naming) — not blocking.
**Notes:** highest-risk change in the project. **REVISIT (Sarunas):** confirm migration on a
copy of the live DB before running for real.

#### Story 8.3 — Event repository + identity service
**Agent:** Opus · **Effort:** medium
Repository + identity service over the unified table; mint `event_id` for all new events;
resolve by `event_id` or `gcal_event_id`.
**Key AC:** create/read/update/delete by `event_id`; publish fills `gcal_event_id` without
changing `event_id`; `has_unpublished_changes` set on local edits, cleared on publish.
**Prereqs:** 8.2.

#### Story 8.4 — Migrate sync reconciler to unified event table
**Agent:** Opus · **Effort:** high
Rewrite GCal sync to reconcile against `event`, matching on `gcal_event_id`, using
`has_unpublished_changes` instead of the old "exists in pending_event" heuristic.
**Key AC:** pull updates published events without clobbering local unpublished edits; new
local approved events publish and bind their gcal id; version-history snapshots still written;
no double-counting between former tables.
**Prereqs:** 8.3.

#### Story 8.5 — Rendering + drilldowns mint candidates in new model
**Agent:** Sonnet · **Effort:** medium
Update all event creation paths (drilldowns' "Create Candidate Event", manual create) and
rendering. `CalendarEventSourceKind` becomes a *derived* view of `lifecycle`/`publish`;
candidates render translucent.
**Key AC:** drilldown candidate buttons create `lifecycle=candidate` events (translucent);
manual create → `approved`; calendar renders candidate/approved/published distinctly; Outlook
removed as a rendered kind (becomes source — see 8.15).
**Prereqs:** 8.3.

#### Story 8.6 — Repoint history/deleted/recurring; remove dead pending code
**Agent:** Opus · **Effort:** medium
Repoint `gcal_event_version`, `deleted_event`, recurring-series references to `event_id`;
delete now-dead `pending_event` code paths.
**Key AC:** version history queryable by `event_id`; delete flow intact; recurring instances
resolve; no references to `pending_event` remain.
**Prereqs:** 8.4, 8.5.
**Notes:** **REVISIT (Sarunas):** recurring events × linking — confirm recurring instances
link independently when recurring editing is tackled.

### Phase 1 — Datapoint registry & coverage

#### Story 8.7 — `data_point` registry table + source-pointer model
**Agent:** Sonnet · **Effort:** medium
Create the `data_point` table (§4): `source_key`, `source_ref`, `start_utc`, `end_utc`. Define
the source-pointer resolution contract (datapoint → raw record / file).
**Key AC:** schema + indexes on `(start_utc)`, `(source_key, start_utc)`; resolver returns the
raw record for a datapoint; instant datapoints use `start == end`.
**Prereqs:** 8.2.

#### Story 8.8 — Import projector contract + guard test + reconciliation sweep
**Agent:** Opus · **Effort:** high
Template-method import base that writes datapoints in the **same transaction** as the raw
insert; a per-source projector contract; a reflection guard test; the reconciliation sweep
(post-import incremental, startup drift-check + heal, manual rebuild).
**Key AC:** a handler cannot import without a projector; CI test fails if any handler lacks
one; startup sweep reports/heals raw records missing datapoints; manual "rebuild registry"
action exists.
**Prereqs:** 8.7.

#### Story 8.9 — Project all existing sources into datapoints (+ natural keys)
**Agent:** Codex · **Effort:** medium
Implement projectors for every existing source (Toggl, calls, Civ 5, ComfyUI, Spotify,
Outlook, …). Add stable natural keys for replace-on-reimport sources (Spotify, Maps) so
`source_ref` survives re-import; cascade-delete datapoints+links on genuine orphan.
**Key AC:** each source backfills datapoints for existing rows; re-import is idempotent (no
duplicate datapoints); removed raw records cascade-delete their datapoint + link.
**Prereqs:** 8.8.

#### Story 8.10 — Coverage service + delete `DateSourceIntegration`
**Agent:** Sonnet · **Effort:** medium
Coverage computation (§6) per (date·source), day, and event → `● / ◐ / ○`, per-datapoint
counts, zero-data ⇒ full. Remove the `DateSourceIntegration` table and its day-mode checkbox UI.
**Key AC:** coverage queries efficient over indexed datapoints+links; zero-data renders as
N/A/done; `DateSourceIntegration` table and write paths deleted; day-mode panel shows computed
coverage instead of manual checkboxes.
**Prereqs:** 8.9, and link table (8.12) for real link state — *may land in two passes (counts
without links first, then link-aware).* 
**Notes:** **MILESTONE (Sarunas):** once this + Epic 9 By-Source lens land, the system is
usable for bulk linking across 3+ sources (source-priority ranking insights start here).

#### Story 8.11 — Block/clump provider contract + adapt existing coalescers
**Agent:** Sonnet · **Effort:** medium
Define the computed **clump → block** provider contract the Linking panel consumes; adapt
existing per-source coalescers (`Civ5SessionCoalescer`, `ComfyUISessionCoalescer`, phone
sliding window) to it. Blocks/clumps are **computed, never persisted**.
**Key AC:** uniform provider returns clumps (datapoint sets) + their blocks (8/15 rounded) for
a source + time range; trivial 1:1 provider for calls/Toggl; existing coalescer outputs match
prior behavior.
**Prereqs:** 8.7.

### Phase 2 — Link model & rule engine

#### Story 8.12 — `link` table + link/ignore/unlink operations (undoable)
**Agent:** Opus · **Effort:** high
Create the `link` table (§5): `data_point_id`, nullable `event_id`, `state`, `origin`,
`rule_id`, `action_group_id`. Implement link / ignore / **unlink (without ignoring)**, all
undoable and grouped by `action_group_id` for clump-level atomic undo.
**Key AC:** unlinked = no row; ignore creates a row with null event; unlink removes the link
row; clump action writes N rows under one group and undoes as one step; cascade-delete on
datapoint removal.
**Prereqs:** 8.7.

#### Story 8.13 — Link-to-any-event picker (tolerates non-concurrency)
**Agent:** Sonnet · **Effort:** medium
Service + selection model to link a datapoint/clump to **any** event (not just concurrent),
to absorb rounding / tracking error.
**Key AC:** picker offers concurrent events first but allows arbitrary event selection; links
written via 8.12; linked/ignored datapoints remain visible in the Linking panel.
**Prereqs:** 8.12, 8.3.

#### Story 8.14 — Rule engine pipeline (propose-ops, invariants, triggers, reversal)
**Agent:** Opus · **Effort:** high
The ordered rule pipeline (§7): rules propose `{link, ignore, generate-candidate}` ops applied
atomically; never override `manual`; deterministic + idempotent; reversal recomputes only
auto state; auditable via `rule_id`. Triggers: import, event approve, event edit (time),
event delete/un-approve.
**Key AC:** re-running the pipeline converges (no oscillation); moving/deleting an event
recomputes auto links and leaves manual links untouched; every auto link records its rule;
full integrity test matrix across operation sequences.
**Prereqs:** 8.12, 8.13.

#### Story 8.15 — First concrete rules: Spotify auto-link, Outlook generate-candidate
**Agent:** Opus · **Effort:** high
Implement the first two rules on the 8.14 engine. **Spotify:** stream covered by exactly one
approved event → auto-link; 2+ overlapping → do nothing. **Outlook:** import proposes
candidate events (Outlook is a raw source; suppressed = ignored datapoints).
**Key AC:** Spotify single-cover auto-links, multi-cover stays unlinked; Outlook import
generates translucent candidates; suppressed Outlook events become ignored datapoints,
accessible in the Linking panel; both reverse correctly when the target event changes.
**Prereqs:** 8.14.
**Notes:** **REVISIT (Sarunas):** this is where the first rules land — review rule behavior and
plan the next batch of the 10+ rule catalog. Also surfaces the (unscoped) rules/automation
**visibility panel** — decide whether to schedule it.

#### Story 8.16 — Closing code terminology-cleanup + naming guard
**Agent:** Codex · **Effort:** low
Catch straggler identifiers not renamed by functional stories (e.g.
`DataSourceDayDataMarkerViewModel`, residual "candidate"/"integration" misuse); add a naming
check so retired terms can't regress.
**Key AC:** no code identifier uses retired data terms for the wrong concept; naming guard
test in place.
**Prereqs:** 8.6, 8.10, 8.15.

## Success criteria

- Single `event` table; every event has a stable `event_id`; publishing never breaks a link.
- GCal sync works against the unified table with no data loss.
- Every data source projects into `data_point`; a new source physically cannot ship without a
  projector (guard test).
- Coverage computes correctly (incl. zero-data ⇒ full); `DateSourceIntegration` removed.
- Link / ignore / unlink are undoable at clump granularity.
- Rule engine holds all integrity invariants across arbitrary operation sequences; first two
  rules (Spotify, Outlook) behave per spec and reverse cleanly.
