# Epic 8 & 9 — Data Linking: Concepts & Data Model

**Author:** Sarunas Budreckis
**Date:** 2026-06-11
**Status:** Design reference (source of truth for vocabulary + data model)

This document is the canonical glossary and data-model reference for the data-linking
work spanning **Epic 8 (Event Model & Raw Data Linking Engine)** and
**Epic 9 (Linking Panel & Workflows)**. All stories in those epics, plus the
terminology doc-sweep, reference this file.

---

## 1. The core idea

The app holds two fundamentally different kinds of information:

- **Raw data** — automatically collected truth (Toggl, calls, Spotify, Civ 5, ComfyUI,
  Maps, Outlook, voice memos, …). High volume, never edited, never deleted.
- **Curated events** — manually audited, interpreted, and colored calendar entries that
  represent *my* understanding of what happened.

The goal of Epic 8/9 is **total accountability**: every raw datapoint is either
**linked** to a curated event or **intentionally ignored** — nothing is left unaccounted.
Two directions of work:

- **Data-first** — "I have raw data; find or create the event it belongs to."
- **Calendar-first** — "I have a gap or an event; find the raw data that fills/corroborates it."

Ideal end state for any moment in time: **0–2 curated events** (usually 1) and
**0–many raw datapoints**, with every datapoint resolved.

## 2. Vocabulary (canonical terms)

| Term | Meaning |
|---|---|
| **Raw data** | The detailed source record(s): a `SpotifyStream` row, a `.m4a` file, a parsed Maps segment. Immortal — never edited or deleted (except a genuine source correction). |
| **Datapoint** | The normalized spine row (table `data_point`) that points to **one section of raw data** (1:1). Carries a time extent + the source pointer. The unit of linking and of coverage counting. Everyday term in UI and code. |
| **Source pointer** | `(source_key, source_ref)` on a datapoint — resolves back to the raw record (a row id, or a file path). Must be non-lossy and stable across re-imports. |
| **Clump** | A set of raw datapoints locally close in time. Lives in the **data domain** (the points themselves). |
| **Block** | The 15-minute-rounded section of **time** a clump occupies (via the 8/15 rule). Lives in the **time domain**. A clump → has a time extent → rounds into one or more blocks. |
| **Candidate event** | An **autogenerated** proposed event (`lifecycle = candidate`), awaiting human approval. Rendered translucent on the calendar. |
| **Approved event** | A curated event a human put on the calendar — by typing it manually (born approved) or by approving a candidate. (`lifecycle = approved`.) |
| **Published** | An event that has a GCal id (`publish = published`). Optional and orthogonal to approval. |
| **Certify (a day)** | The day-level review layer ("this day is reviewed"). Independent of event approval and of linkage completeness. |
| **Link** | A resolution row binding a datapoint to an event (`state = linked`) or marking it dismissed (`state = ignored`). Absence of a link row = **unlinked**. |
| **Coverage** | For a scope (date·source, or event): the fraction of in-scope datapoints that have a link row (linked **or** ignored). |
| **Rule** | Hardcoded automation that proposes operations: `link`, `ignore`, or `generate-candidate`. |
| **Link order** | A ranked source list of **optional, non-blocking** dependencies ("finish Toggl before Spotify") that drives Linking-panel ordering and soft prompts. *Not* a rule — performs no operations. |
| **Linking panel** | The third left panel (Epic 9) hosting all four linking workflows. |

**Retired terms:** "integration" / `DateSourceIntegration` (replaced by computed **coverage**);
"artifact" (replaced by **datapoint**). Do not reuse these for data concepts.

## 3. Event model (unified table)

Epic 8 Phase 0 **merges** today's `gcal_event` and `pending_event` into a single `event`
table keyed by a **stable local id**. This is the single biggest schema change in the
project and the foundation for link integrity.

```
event
  event_id              PK, stable local string — minted for EVERY event, never changes
  gcal_event_id         nullable, UNIQUE — filled on publish; sync matches on this
  calendar_id
  summary, description, start_datetime, end_datetime, is_all_day, color_id
  lifecycle             'candidate' | 'approved'
  publish               'local_only' | 'published'
  has_unpublished_changes  bool   — replaces the old "row exists in pending_event = dirty" heuristic
  source_system         'manual' | 'toggl' | 'outlook' | 'civ5' | ...   (origin)
  recurring_event_id, is_recurring_instance
  gcal_etag, gcal_updated_at, last_synced_at
  app_last_modified_at, created_at, updated_at
```

**Why a stable `event_id`:** links always reference `event_id`. Publishing a local event
fills `gcal_event_id` but `event_id` never changes — so **no link ever breaks on publish.**

### Lifecycle

```
                 (manual create)                (approve)
                       │                            │
   raw data ──rule──► candidate ───────────────► approved ──(push)──► published
                       │  (translucent)             │                   (gcal_event_id set)
                  reject = delete              edit / move / delete
```

- **Manual create** → born `approved`.
- **Rule generate-candidate** (e.g. Outlook import) → `candidate`, translucent, awaiting approval.
- **Delete any event** → its datapoints become **unlinked** (raw data persists; see §6).
- **Day certification** is a separate axis over a *date*, not an event state.

### Migration (full data preservation — all rows transformed, only `DateSourceIntegration` discarded)

All event data is kept and transformed; nothing is dropped except the manual per-day "integrated?"
checkbox rows in `DateSourceIntegration` (replaced by computed coverage in Epic 8 §6).

- each `gcal_event` → `event(approved, published)` — all event content preserved
- overlay `pending_event` (has gcal id) → **merge into** that event row, set `has_unpublished_changes`
- manual `pending_event` (no gcal id) → `event(approved, local_only)` — content preserved
- machine `pending_event` (`source_system ≠ manual`) → `event(candidate, local_only)` — content preserved
- existing `LinkedEventId` (gcal-id strings) rewritten to stable `event_id`
- repoint `gcal_event_version`, `deleted_event`, recurring series FKs to `event_id`
- `DateSourceIntegration` rows → **discarded** (manual checkbox; superseded by coverage)

## 4. Datapoint registry (the linking spine)

```
data_point
  data_point_id   PK
  source_key      string   — which source ('spotify_stream', 'toggl_entry', 'voice_memo', ...)
  source_ref      string   — pointer to the raw record (row id or file path)
  start_utc, end_utc        — time extent (instant ⇒ start == end)
  created_at
```

- **1:1 with a raw record.** Every importable raw record projects into exactly one datapoint.
- **No link state stored here** — link state is the link table (§5). Unlinked = no link row.
- **Re-import orphan = cascade.** If a re-import proves a raw record genuinely no longer
  exists, its datapoint **and** its link are deleted. (Natural keys for Spotify/Maps reduce
  false orphans.)

### Projection enforcement (so future handlers can't "forget")

1. **Template-method import base** — the base class writes datapoints in the **same
   transaction** as the raw insert; a handler can't import without supplying its projection.
2. **Reflection guard test** — CI fails if any `IDataSourceImportHandler` lacks a projector.
3. **Reconciliation sweep** — backstop that detects raw records lacking datapoints.
   Runs **post-import** (incremental), **on app startup** (cheap drift check + heal), and
   via a manual **"rebuild registry"** action.

## 5. Links

```
link
  link_id          PK
  data_point_id    FK → data_point  (ON DELETE CASCADE)
  event_id         FK → event, nullable (null when state = 'ignored')
  state            'linked' | 'ignored'
  origin           'manual' | 'auto_rule'
  rule_id          nullable — which rule set it (auditability)
  action_group_id  string   — groups a multi-datapoint operation for atomic undo
  created_at, updated_at
```

- **Unlinked** = datapoint with no link row.
- **Link / ignore / unlink** are all **undoable**, grouped by `action_group_id` so a
  clump-level action undoes as one step.
- **Unlink ≠ ignore.** Unlinking returns a datapoint to the unlinked pool; ignoring
  resolves it (dismisses the task) while remaining accessible in the Linking panel.
- **Link to any event** — not restricted to time-concurrent events (tolerates rounding /
  tracking error). The user may pick any existing event.
- `origin`/`rule_id` distinguish automation from manual decisions (see §7).

## 6. Coverage

For a scope — a **(date, source)** cell, a **whole day**, or a **single event**:

- Gather the in-scope datapoints.
- **Covered** = has a link row (linked **or** ignored).
- `● full` = all covered · `◐ partial` = some · `○ none` = zero covered.
- **Zero datapoints in scope ⇒ full coverage** (shown as "N/A / done / —", not a scary 0/8).
- Counters are **per datapoint** ("12/40 datapoints linked").

Coverage **replaces** the deprecated manual `DateSourceIntegration` checkbox entirely.

## 7. Rules vs Link order

### Rules (operations)

Hardcoded automation. A rule proposes operations — `link`, `ignore`, or
`generate-candidate` — never writes directly. Invariants:

- **Manual decisions are sacred.** Rules only touch datapoints that are unlinked, or that a
  rule itself previously set (`origin = auto_rule`). Never override a `manual` link/ignore.
- **Deterministic + idempotent.** Same inputs → same outputs regardless of import order;
  re-running the full pipeline converges.
- **Reversal recomputes only auto state.** When an event is moved/deleted, rule-set links
  recompute; manual links are untouched.
- **Auditable.** Every auto-link records its `rule_id` ("why is this linked?").
- **Atomic.** A rule pass runs in one transaction.
- **Per-rule tiebreak.** Each rule defines its own behavior when multiple events overlap.

**Triggers:** data import · event approval · event edit (time change) · event delete/un-approve.

First two concrete rules (Epic 8):
- **Spotify auto-link** — a stream covered by exactly **one** approved event → auto-link;
  **2+** overlapping approved events → do nothing (leave unlinked).
- **Outlook auto-generate-candidate** — Outlook import proposes candidate events (Outlook is
  a raw source, no longer a rendered event kind).

### Link order (sequencing, non-blocking)

A ranked source list of **optional dependencies** ("finish Toggl before Spotify"). Performs
no operations. Drives By-Source ordering in the Linking panel and optional soft prompts —
**never blocks** the user. Hardcoded for now. This is the "natural order / ranking."

## 8. The four linking workflows (Epic 9)

Hosted in the **Linking panel** (3rd left icon, alongside Sources and Day Detail):

- **By Source (W1+W2, merged)** — pick/iterate a source; see its unlinked clumps; link /
  ignore / unlink / +event; Next/Prev. Scope = view-following **or** an independent date
  range. Source list is driven by **rule membership**: every source named in the hardcoded
  rules appears (even with no data), showing `x/y linked`; sources not in any rule sort to
  the end and appear only if they have data.
- **By Event (W3)** — for a selected event (or right-click → "show concurrent raw data"),
  list all time-overlapping datapoints grouped by source; bulk link/ignore; show event coverage.
- **Gaps (W4)** — raw data in scope **not** covered by any approved event, clumped
  cross-source by blank period; each gap → create-event-from-gap (→ translucent candidate)
  or ignore. Gaps are always ≤ 24h. When the Gaps lens is active, the calendar renders empty
  gray outlines with a `+` in place of events; clicking a gap opens a **gap detail panel**
  (top-4 sources, vertical color-coded dots, instant tooltips, expandable for high-volume).

## 9. Source-specific notes

- **Outlook** — raw source that **auto-generates** candidate events on import; "suppressed"
  Outlook events = ignored datapoints (still accessible in the Linking panel). No longer a
  rendered `CalendarEventSourceKind`.
- **Maps Timeline** — currently a raw JSON blob; an Epic 7 story parses it into per-segment
  datapoints (using the Google Maps API, as the standalone viewer does).
- **Voice memos / Chrome search** — link/ignore **only** for now (no generate-candidate),
  because one clump often spans several events and we lack a programmatic boundary-extraction
  method. **Future:** if/when boundary extraction exists, allow generate-candidate here too.
- **Spotify / Maps re-import** — replace-on-reimport sources need **stable natural keys**
  (e.g. Spotify `PlayedAt + TrackName`) so datapoint `source_ref`s survive re-import.
- **Blocks/clumps are computed, never persisted** — a clump is a named selection of
  datapoints; linking a clump writes N per-datapoint links under one `action_group_id`.
