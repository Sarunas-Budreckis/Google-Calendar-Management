## What Went Well

**1. DI container established from Story 1.1**
Setting up the dependency injection container on day one paid compound interest across every subsequent story. `MigrationService`, `LoggingService`, and `ErrorHandlingService` all dropped into the container without rewiring. No retroactive refactoring.

**2. Real SQLite integration tests, not in-memory**
Using temp-file SQLite in integration tests caught real issues that in-memory SQLite would have silently passed: the WAL mode interceptor requirement, `SqliteConnection.ClearAllPools()` cleanup, and the backup-before-migration file I/O path. These would have been production mysteries.

**3. Backup-before-migration pattern**
Simple in implementation, critical in correctness. Protects user data on every startup. The right call not to defer it.

**4. Logging configured before DI setup**
Placing `LoggingService.Configure()` at the very top of the `App()` constructor — before `ConfigureServices` — means any DI setup failure is captured in the log file rather than disappearing silently. A subtle but important sequencing decision.

**5. Fluent API only for EF Core entities**
Enforcing Fluent API (no data annotations) kept entity classes clean and configuration consistent across all 7 tables. No exceptions crept in.

**6. Clean infrastructure metrics**
245ms launch-to-window, 419ms migration, integrity check every startup — all well within design targets.

## What Was Harder Than Expected

**1. Serilog package ecosystem drift (Story 1.6)**
Three deviations from the spec required investigation:
- `Serilog.Sinks.Console` 6.0.1 not on NuGet; 6.1.0 used
- `WriteTo.TextWriter` removed from Serilog 4.x core; required custom `InMemorySink`
- `SerilogTimings` does not expose `BeginTimedOperation`; use `OperationAt(...)` instead
- `services.AddSerilog()` on `IServiceCollection` requires `Serilog.Extensions.Hosting` (not installed); use `services.AddLogging(builder => builder.AddSerilog())` instead

**2. Platform=x64 requirement**
MSIX packaging requires explicit platform specification for every build and test command (`-p:Platform=x64`). Not obvious from the WinUI 3 template. Any developer new to this project will hit a confusing failure without that flag.

**3. WAL mode via interceptor, not connection string**
`Journal Mode` is not a valid Microsoft.Data.Sqlite connection string keyword. WAL mode must be set via raw `PRAGMA journal_mode=WAL` inside an EF Core interceptor. Initial assumption was wrong; took investigation to resolve.

**4. ContentDialog XamlRoot timing (Story 1.4 / 1.6)**
`ContentDialog` requires `XamlRoot` to be set, which only becomes available after `window.Activate()`. The window must be activated before calling `RunStartupAsync()` and before registering the global exception handler's user-facing dialog. Discovered late; required startup sequence adjustment.

## Conventions Established in Epic 1

The following conventions are now project-wide standards:

- **Story notation:** `Story 1.5` in prose; `1-5-story-name` in filenames and YAML keys
- **Project delivery milestones:** "Tier 1 / 2 / 3" (not "Phase") to avoid collision with BMAD workflow phases
- **Folder structure:** `docs/epic-{N}/` for tech spec and retrospective; `docs/epic-{N}/stories/` for story files and context XMLs
- **Platform flag:** All `dotnet build` and `dotnet test` commands require `-p:Platform=x64`
- **Logging:** `Log.Logger` configured before DI; `Log.CloseAndFlush()` on every exit path
- **EF Core:** Fluent API only, no data annotations; WAL mode via interceptor
- **Tests:** Integration tests use real temp-file SQLite, never in-memory for file I/O paths
